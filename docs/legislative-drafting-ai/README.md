# 立法・議会活動サポートAI ― 構想メモ（v0.1）

> 地方議会議員の立法・政策活動を支えるAIシステムの、最初の一枚。
> 「議会事務局のプロフェッショナルなサポートが、人員的に成り立たなくなっている」
> という現場課題を、AIで補完することを狙う。

作成日: 2026-07-13 / ステータス: 構想（未着手）

---

## 1. なぜやるか（背景・課題）

- 地方議会議員が政策・立法活動を行うには、本来は**議会事務局の専門的サポート**（法制執務、先例調査、資料作成、想定問答の準備など）が必要。
- しかし多くの自治体で事務局は**慢性的な人員不足**にあり、一人ひとりの議員に十分な時間を割けない。
- 結果として、議員個人の情報収集力・調査力に活動の質が依存してしまう。
- ここを**AIで補完**できれば、少人数でも質の高い政策活動が可能になる。

## 2. コンセプト（今回の割り切り）

**「議員個人のための調査・起案アシスタント」として作る。**

| 論点 | 今回の立場 | 理由 |
|---|---|---|
| 誰のためか | **議員個人**（まずはながとり太郎氏の実務） | 中立性・全会派対応の制約を外し、小さく速く検証できる |
| 中立性 | 不要（個人ツール） | 特定議員の問題意識・スタンスに最適化してよい |
| 事務局代替か | 代替ではなく**議員側の武装** | 事務局の合意形成なしに始められる |

> 将来的に事務局向け（全議員・全会派に中立提供）へ展開する余地は残すが、
> **v1では個人ツールに徹する。**

## 3. 「立法支援」を業務に分解する

「条例案を書く」という最も重い部分を、いきなり狙わない。頻度・リスクで分解する。

| 業務 | 頻度 | 事務局依存度 | AI適性 | リスク | 優先度 |
|---|---|---|---|---|---|
| 一般質問・政策提言の準備 | **高**（毎定例会） | 中 | 高 | 低 | ★★★ |
| 過去質問・答弁の検索/要約 | 高 | 中 | 高 | 低 | ★★★ |
| 他自治体の条例・先進事例調査 | 高 | 高 | 高 | 低 | ★★ |
| 議員提案条例の起案・法制執務チェック | 低（数年に一度） | 極高 | 中 | **高**（法的責任） | ★ |

**判断：** 議員提案条例の自動起案は最終ゴールとしては魅力的だが、
**低頻度・高リスク・事務局の専門性が最も凝縮**しているため、とっかかりには向かない。
最も頻度が高く低リスクな **「一般質問・政策提言の準備支援」** を最初の楔（くさび）にする。

## 4. とっかかり：一般質問・政策提言の準備支援

汎用チャットAIに対する決定的な差別化は、
**「この議員・この自治体の文脈を、AIが既に知っている」** 状態を作れること。
そしてその素材は**既にこのリポジトリにある**（→ 第5節）。

### 想定フロー

```
入力：議員が扱いたいテーマ（例：「学校体育館へのエアコン設置」）
   │
   ├─ ① 過去ログ横断検索 ─────────────────────────
   │     自分/同僚の過去質問・執行部答弁を拾い、
   │     「どこまで議論済みか・執行部の従来スタンス」を提示
   │     （＝同じ質問の繰り返しを防ぐ。事務局が頭の中でやっていること）
   │
   ├─ ② 先進事例・国の動向を調査（Web） ──────────
   │     他自治体の条例・施策、国のガイドライン等を収集・比較
   │
   ├─ ③ 質問ドラフト生成 ───────────────────────
   │     「現状認識 → 課題 → 具体的提案 → 執行部への問い」の型で草案
   │
   └─ ④ 想定答弁・再質問の準備 ─────────────────
         執行部が返しそうな答弁と、それへの切り返しを用意
         （過去答弁のパターンから「はぐらかし」を予測）
```

**最小の第一歩は ①**。データが既にあり、事務局の「代わり」を最も実感できる。

## 5. 既にある資産（このリポジトリ）

このリポジトリは単なる議事録アーカイブではなく、**構造化済みの一次データ**である。

- 一般質問・本会議・委員会の議事録（HTML）
- 各質問に **「質問の要旨」「市側答弁の要旨」** の要約が付与済み
- 動画への**タイムスタンプ付き文字起こし**
- 例：`20260224_nagatoritaro_shitsumon.html` … 2040年問題／デジタルレイバー、
  バリアフリー情報の可視化、街頭消火器 など

→ **これをそのまま検索・要約の対象データにできる。** データ整備の初期コストが低い。

## 6. MVP仕様（v1 の最小形）

**目的：** テーマを入れると、関連する過去質問・答弁を返す「議員専用の記憶」。

- **入力：** 自然文のテーマ／キーワード
- **処理：**
  1. 既存HTMLをパースし、`{日付, 会議種別, 議員, テーマ, 質問要旨, 答弁要旨, 全文, 動画URL, タイムスタンプ}` の構造化データに変換
  2. 全文＋要旨をインデックス化（まずは全文検索、次段でベクトル検索/RAG）
  3. クエリに対し、関連質問と**執行部の従来スタンス**を要約して提示
- **出力：** 「このテーマは R8.2.24 に質問済み。執行部は『研究を進める』と答弁（＝実質保留）」のような**論点の履歴**
- **非目標（v1でやらないこと）：** 条例起案、法制執務チェック、全会派対応、リアルタイム議会運用

### 技術メモ（案・確定ではない）
- データ変換：HTMLパース → JSON（`data/` に蓄積）
- 検索：まず素朴な全文検索で価値検証 → RAG（埋め込み＋LLM）へ
- LLM：要約・草案生成は Claude を想定
- UI：まずCLI/簡易Webで内部検証、公開UIは後

## 8. 先に潰しておきたい論点・リスク

- **データの網羅性と鮮度：** 議事録がPDF/HTMLでバラつくと、検索精度が全部そこで決まる。**データ整備が地味だが本丸。**
- **ハルシネーション：** 政策の場でAIが事実を捏造すると致命的。**必ず一次資料（議事録の該当箇所）へのリンクを添える**設計にする。最終判断は議員。
- **他自治体データの権利・利用範囲：** Web調査で拾う条例・資料の引用範囲に注意。
- **将来の中立化との整合：** v1は個人最適でよいが、事務局展開時に「特定議員色」が足かせにならないよう、データ層と議員個人の設定層は分けておく。
